OPG - SIRIUS - SUPERVISION

The project

The Office of the Public Guardian (OPG) is undertaking a transformation project to improve the user experience and the accessibility of their case management systems.

This part of the project is to design a workflow tool to help Workflow Managers track and share tasks within, and between, their teams.

Workflow Managers export data for each team’s workload and a snapshot file is manually updated every week. This file covers the type of tasks, what has been completed, what is due and the age of due tasks. These tasks depending on the date are colour-coded to highlight those tasks that are overdue.

This helps Workflow Managers to see what work needs to be completed in a given week, prioritise overdue and important tasks, and reassign tasks.

Each workflow manager has a different process to allocate work to their teams because there is no standard process or tool.

Understanding the current processes

I had joined OPG from HMPPS and Central Digital so was keen to see how things were currently done. I began shadowing some of the Workflow Managers. This allowed me some more casual time for myself to watch users complete tasks and to ask questions.

I then paired with the User Researcher for a series of more formal interviews with the Workflow Managers, this was great in adding further knowledge and filling in the gaps.

State of play

I joined the team after it had gone through a long period of design hiatus. This meant that the designs I inherited were mostly in need of some updated research and iteration. The live system however was in a state of limbo with large parts of the legacy system being slowly replaced.

The user researcher on the team was also new so it was a great opportunity for us to really learn the space together.

Before we went into testing I wanted to update some of the patterns in the prototype and test those. The due date highlights the urgent date with a coloured line and states clearly underneath what the detail of the urgency is. I felt as though the line shown was too thin and thought we could adopt the MOJ tag pattern

Time to test

Testing was done with 3 different teams - Lay, Pro and Public Authority initially. Each of these teams have a need to see different items of data so I created these different views to help in testing. The first round of testing was done with a member from each team.

Round 1 of testing

Some things we wanted to validate with the HTML prototype

  • Is the information within the table relevant?
  • Did they find the data in the grey “your tasks” section useful?
  • Were the filters useful?
  • Did users understand the tags? Was it still obvious what this feature was communicating to them?

Round 1 of testing results

Some things we wanted to validate with the HTML prototype

  • The information around the tasks within the table was useful and nothing else was needed
  • The data in the “your tasks” was not all that helpful and took up valuable space on the page
  • The filters provided information that was useful - however there was feedback on certain processes still remaining long winded.
  • The tags were understood and preferred the new layout - users commented that the addition of more colour, combined with the status made it easier on the eye. It also conformed with updated GDS pattern guidance.

The main identified problem for Workflow Managers came when checking staff workload. Users would have to select the task type and then the colleague. This had the potential to take a while to load. When taking into account users were repeating the process then with the next person and so on, it became very time consuming.

Round 2 of testing

Before we did the second round of testing I made some iterations:

  • Removed the grey data box at the top, most users just commented on how much space it took up
  • Added in numbers indicating caseloads for staff - this was so Workflow Managers could get a quick snapshot of their Case Managers workload.
  • Added deputy into the table, was this something users found useful

Round 2 of testing

This was a quick follow up session to test some iterations and improvements from the previous session:

  • Did the users understand the proposed solution for the problem of time consuming task allocation?
  • Did users understand the proposal for allocating tasks?

Round 2 of testing results

  • The solution proposed was to place the number of tasks next to the individual in the filter list. This solution was received positively as users instantly could see how much time it would save.
  • Users also understood the process of selecting multiple tasks and then undertaking the steps in successfully reallocating those.

Outcome

The result of this design and iteration cycle is the workflow tool offers a more standardised and consistent way of working. For workflow managers it will reduce the time exporting data into spreadsheets, creating reports and then allocating work. This work now allows users to see the workloads of their staff at a glance and then offers the user the opportunity to allocate right there. This has a further knock-on effect that the users are able to focus their attention into other tasks.
